# Course Authoring Playbook

Distilled from building the **Customer Support Bot** course end-to-end
(and the prior Finance RAG / Wikipedia QA post-mortems). Follow this to
create an outcome-mode course that is *fair, passable by the taught
technique, ungameable, and cheap to validate.*

---

## 0. The cardinal rule

> **The green/pass bar must be reachable by the technique the course
> teaches, using only the tools the course says are required — with NO
> dependence on an LLM judge.**

Finance RAG and Wikipedia QA shipped *unpassable*: an `llm_judge`-gated
bar against terse single gold (and a no-LLM/offline constraint) meant a
*correct* solution still failed. Every rule below serves this one.

- Gating rubrics → **deterministic only** (`schema_match`,
  `literal_match`, `numeric_range`, `subset_match`, `oracle_set_overlap`).
- `llm_judge_*` rubrics → **bonus, non-gating, lenient, with `alt_ans`**.
- Prove the bar is reachable by grading a real reference before ship.

---

## 1. Course shape

Outcome-mode course = one HTTP service graded by hidden scenarios.

- Define an unambiguous **contract** (request/response JSON, field types,
  enums, thresholds) in `project_brief.md`. Multi-turn → include
  `conversation_id` + `history`.
- Bundle layout: `private/grader/{scenarios/*.yaml,_setup/*.json,
  _reference/app/main.py}` + `public/starter/...`. The course run's
  `payload_json.outcome_state.workspace_root` points at this dir; embed
  the bundle in the publish snapshot (don't rely on the mutable
  authoring workspace — that was a live P0).

## 2. Skills-grounded design (traceability, not vibes)

Build a table BEFORE writing scenarios: **Skill → subskills → required
tool(s) → which rubric grades it → gating or bonus**. Every scenario
must trace to a skill. No scenario without a skill; no skill without a
gating deterministic test (except the explicit bonus skill).

## 3. Reference implementation rules

- Write a **proven reference** whose *gradeable decisions are pure
  deterministic code* (policy-as-code, regex, IR) — no LLM in the
  decision path. The LLM (if any) only polishes prose and must degrade
  gracefully (offline → templated, never crash, never block a decision).
- Structure it as a pure `decide(req) -> resp` core + thin HTTP layer so
  you can unit-test every decision locally at **$0** (no FastAPI/LLM).
- The reference must score **full green by construction** (see §4).
- It must model *good* behavior: e.g. emit `citations: []` for
  non-answer actions (don't let retrieval noise become "gold").

## 4. Scenario + gold authoring (the part that bites)

**Gold is generated by running the proven reference** over each
scenario → guarantees the reference passes and there are no contradictory
same-input pairs. BUT generating gold from the reference *over-fits to
the reference's quirks* unless you defend against it:

| Trap (real, found this run) | Rule |
|---|---|
| `oracle_set_overlap` recall-only → "cite everything" passes | Pair recall with **precision**: `subset_match` `acceptable_source` = the **gold set**, not the whole input pool. Cited ⊆ gold AND recall ≥ 0.5. |
| `literal_match` on an exact count/value → a *more correct* impl that counts differently fails | Grade a **spec-derived bound or category**, not the reference's exact number. e.g. PII → `numeric_range min=#categories-by-spec, max=null` (≥), tool-agnostic. |
| Reference's retrieval noise enshrined as gold on escalate/refuse | Reference emits `citations:[]` for non-answer actions; never put a citation rubric on a non-answer scenario (`subset_match` also *fails on empty target*). |
| Softball negatives (weather/recipe) don't test the skill | Add **near-domain** distractors (e.g. a privacy-exfil request that *sounds* like support but must be refused) and **paraphrased** adversarial inputs (not just the canonical "ignore previous instructions"). |
| Over-fit single gold + `llm_judge` gating | `llm_judge_*` bonus only; `strictness: lenient`; populate `alt_ans`. |

`Scenario.category` is a **fixed enum** (`happy_path, boundary,
malformed_input, adversarial, out_of_scope, idempotency, determinism,
concurrency, state_persistence, composition`). Map skill tags onto these
or scenario load 500s. Make the generator **idempotent** (clear the
scenarios dir before emit) so renumbering can't leave duplicate ids.

## 5. Rubric selection cheatsheet

- **schema_match** — contract conformance (gating, every scenario).
- **literal_match** — exact enum/bool (`action`, `abstained`). Never for
  fuzzy/count values.
- **numeric_range** — counts/scores with a fair `>=`/range bound.
- **oracle_set_overlap** + **subset_match(gold)** — citation recall +
  precision. Always pair them.
- **regex_match** — format (uses `re.fullmatch`, whole-string).
- **llm_judge_semantic_eq / coverage / false_premise** — quality/semantic;
  **bonus only**, lenient, `alt_ans`.
- **behavioral_equivalence** — stability under perturbation (decide via
  `expected`/`expected_path`).

## 6. Honesty in the brief

Don't name-drop tools the grader doesn't reward. If lexical retrieval
clears the bar (small per-request KB), say embeddings/FAISS are the
*illustrative production-scale path, not required*. State plainly what
the course does and does NOT teach (e.g. "grounded-answering discipline,
not retrieval engineering at scale").

## 7. LLM-in-course infra (if learners may call an LLM)

The grader sandbox blocks egress. To allow runtime LLM safely:

- A **platform proxy** (Haiku-only) with a **per-submission token cap**
  AND a **global USD hard-stop** (`LAB_LLM_GLOBAL_USD_CAP`); logs every
  call's tokens+cost.
- `workspace_boot` opens egress **only** to the proxy (host-gateway) and
  injects a short-lived scoped token; mint per submission, **revoke in
  `finally`**. Additive/no-op for courses that don't use it.
- Grade **decision stability**, not LLM text (temp 0; idempotency checks
  `action`/`citations`, not `reply` bytes).

## 8. Validation workflow (do this order — it's cheap and catches bugs)

1. **Local, $0:** unit-test the reference's pure `decide()` across every
   skill; generate the bundle; structurally validate YAML.
2. **Server, $0:** `scenario_loader.load_scenarios_from_dir` +
   `_load_setup_data` — catches enum/duplicate/path errors **before** any
   graded run (caught two 500s this way, zero spend).
3. **Graded, budgeted:** baseline (starter → low score, proves grader
   discriminates) → reference (→ full green, proves passable) →
   **adversarial probe** (cite-all / over-count / injected → proves the
   ungameable + tool-agnostic claims). Read the logs; distinguish a real
   low score from a 502/disk/build failure.
4. **Cost discipline:** Haiku tier; local stub for reference iteration
   (no real LLM until logic is correct); cap graded runs; watch the
   spend log; the proxy hard-stops regardless. Whole Support Bot build +
   ~7 graded runs cost **< $0.03**.

## 9. Learner feedback quality (the worked-example panel)

For every failed scenario the learner must see: the **actual input**,
the **expected value**, **their output**, the **precise failing check**,
and a **skill-correct concept hint**. Verified requirements:

- Question extraction must be **field-agnostic** (`message`/`question`/
  `prompt`/…), not hardcoded to one contract.
- Expected/label must bind to the **specific failing rubric instance**
  (not first-of-kind) — else multi-rubric scenarios show contradictory
  "Expected".
- `Expected` resolution must cover **every** rubric kind's gold/threshold
  key (not just `gold_path`/`expected`) — else blank Expected.
- Hints keyed on **kind + target** (`literal_match on action` ≠ `on
  redactions`).
- Lock it with regression tests (`tests/test_worked_example.py`,
  incl. a guard that fails if a new rubric kind is uncovered).

## 10. Ops hygiene

- Deploy via `rsync` + service restart; **never hand-edit prod outside
  git**; everything reversible.
- Each graded submit writes a 1–2.5 GB throwaway image → run the
  **disk-prune systemd timer** (`scripts/grader_disk_prune.sh`); disk
  filled to 100% mid-run once.
- Back up any DB payload before editing (course/snapshot edits).
- "Check logs always; verify before spend; don't hand-wave."

## 10b. Honest limitations — document, don't over-engineer

A deterministic, no-LLM grader whose gold is derived from a heuristic
reference has *inherent* limits. State them; don't paper over them.

- **Citation precision is reference-tie-break-overfit.** `subset_match
  min_overlap: 1.0` vs the gold set means a learner must cite *exactly*
  the reference's chosen supporting subset. A learner who cites a
  *defensible superset* (an extra genuinely-supporting article) false-
  fails. This is the unavoidable flip side of killing the "cite-all"
  false-pass — precision vs. over-fit. Mitigation: keep S1/S6 queries
  unambiguous (one clearly-supporting doc per query) so the gold subset
  is uncontroversial; only loosen to a Jaccard/precision-floor if
  scenarios become genuinely multi-doc.
- **Regex adversarial/OOS guards test "the reference's regex fires,"
  not real robustness.** Keyword `_PRIVACY_EXFIL_RE`/`_INJECTION_RE`
  guards (and the 1–2 scenarios shaped to hit them) are *illustrative*
  coverage. A brittle learner impl using the same shallow regex passes;
  a real adversary (paraphrase, non-English, semantic) is not tested. A
  genuine robustness bar needs semantic detection — out of scope for a
  no-LLM deterministic course. Say so in the brief (same honesty rule
  as retrieval tooling).

These are acceptable for a *teaching* grader; they are not a credible
*production-robustness* bar. Be explicit about which one you're shipping.

## 10c. Learner-facing brief discipline

- **Author for the actual LMS markdown renderer** (`lms.js
  renderMarkdown`): it supports only headings-from-bold, paragraphs,
  `- ` bullet lists, and inline `code`/**bold**. **No pipe tables, no
  fenced code blocks, no numbered lists** — they render as run-on text.
  A GFM table in the Skills/contract section shipped broken.
- **Fixed, fully pre-installed toolset; forbid learner deps edits.** A
  "comment out / uncomment to use" requirements pattern means any learner
  edit busts the Docker deps-cache → a multi-minute reinstall on *every*
  submission. Pin the complete endorsed set, install it all once, bake
  any model it needs at build time, and tell the learner in the brief +
  a Dockerfile guard banner: *use only these; do not edit requirements
  or the deps step.* Deliberately exclude genuinely heavy deps
  (spaCy/large HF models) and say why.
- **Keep the brief actionable; quarantine production context.** State
  what the learner must DO plainly (e.g. "S8 LLM is capped at ~60k
  tokens/submission — at most 1–2 short calls; fall back on failure").
  Production-only material (observability/Langfuse/eval) goes in a
  clearly-labeled *"context only — nothing to build here"* section, not
  woven into instructions. Ship a ready helper for any non-trivial
  integration (e.g. a `call_llm()` stub) instead of dumping the raw
  HTTP/JSON contract on the learner. Don't include a "local dev"
  walkthrough that can diverge from the graded path and confuse.

## 11. Pre-ship checklist

- [ ] Skill→rubric traceability table exists; every scenario maps to a skill.
- [ ] Reference scores **full green**; gradeable decisions are LLM-free.
- [ ] Citation skills test **precision AND recall** (cite-all fails).
- [ ] Count/score skills use a **tool-agnostic bound** (no exact `==`).
- [ ] No `llm_judge_*` on the gating path; bonus ones lenient + `alt_ans`.
- [ ] No contradictory same-input scenarios; non-answer ⇒ no citations.
- [ ] Near-domain + paraphrased adversarial scenarios present.
- [ ] Brief honestly states required vs illustrative tools, and what's
      out of scope.
- [ ] `scenario_loader` + `_load_setup_data` pass server-side ($0).
- [ ] Baseline-low / reference-green / adversarial-probe verified.
- [ ] Worked-example feedback covers all rubric kinds; hints correct.
- [ ] Bundle embedded in the publish snapshot; prune timer active.
